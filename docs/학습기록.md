## 1. 프로젝트가 하는 일

공 위치와 홀컵 위치를 탭하면, 브루트포스 없이 "이 방향/속도로 치면 홀인한다"는
(direction, speed) 조합의 집합을 계산해서 화면에 보여준다.

핵심 아이디어: 홀컵에서 시작해서 시간을 거꾸로 돌리며(backward) 공 쪽으로
거슬러 올라가 초기 후보를 빠르게 만든 다음, 그 후보를 실제 정방향(forward)
시뮬레이션으로 재검증하고 오차를 반복 보정한다. 브루트포스로 모든 방향/속도
조합을 다 시도하는 대신, 홀컵→공 역추적으로 후보를 좁혀서 계산량을 줄이는 접근.

## 3. 핵심 개념: dt의 부호와 forward/backward

`GolfBall.updateFromTorque(deltaTime:surfaceNormal:)` 하나의 함수를 두 곳에서
재사용한다:

- **정방향(forward)** — 실제 게임플레이, `verify()`의 재검증: `dt = +config.deltaTime`
  (양수). 시간이 실제로 흐르는 정상적인 물리 계산.
- **역방향(backward)** — `backwardCandidate()`의 홀컵→공 역추적: `dt = -config.deltaTime`
  (음수). "시간을 거꾸로 돌려" 공이 어디서 왔는지 역산하는 트릭.

**"가파른 내리막"은 forward도 backward도 아니고, 그 지점 지형 자체의 고정된
성질**이다 — 중력 방향 기준으로 그 경사면이 얼마나 가파른 내리막인지는 공이
어느 방향으로 지나가는지와 무관하게 정해진다. 반면 "포워드에서 내리막이면
가속, 백워드로 훑으면 감속"이라는 건 **같은 경사를 어느 순서로 훑느냐**의
문제 — 서로 다른 축의 개념이므로 혼동하지 않도록.

## 4. 버그: 역방향 적분이 정방향의 정확한 역연산이 아니었음 (커밋 fb54818)

### 증상

`backwardCandidate`가 실제로 존재하는 해를 찾지 못하고 `nil`을 반환하거나,
경사가 가파를 때(대략 2.9도 / 5% 경사 이상, 실제 그린에서 흔한 정도) 계산이
발산했다.

### 원인

`updateFromTorque`는 매 스텝마다 순서대로 3가지를 한다:
1. 중력이 공을 민다 (`velocity += acceleration * dt`) — 항상 고정된 내리막 방향
2. 구름저항이 속도를 깎는다 (`applyRollingResistance`)
3. 그 결과 속도로 위치를 이동시킨다 (`position += velocity * dt`, **이 시점의
   속도**, 즉 1·2단계를 거친 후의 값을 사용)

정방향에서는 이 순서(밀고→깎고→그 결과로 이동)가 물리적으로 맞다. 하지만
역방향("영상을 거꾸로 되감기")이 진짜로 정확하려면, 단순히 `dt` 부호만
뒤집어서는 안 되고 **순서 자체와 위치 계산에 쓰는 속도값도 뒤집어야** 한다:

- forward: (가속 → 마찰 → 그 결과 속도로 위치 이동)
- backward(정확한 역연산): (그 스텝 **진입 시점** 속도로 위치를 먼저 되돌림 →
  마찰을 되돌림 → 가속을 되돌림)

기존 코드는 `dt` 부호만 바꾸고 순서와 위치 계산에 쓰는 속도값은 그대로
뒀다 — 그 결과 한 스텝 되감을 때마다 미세한 오차가 쌓였고(누적 오차),
경사가 구름저항보다 강하면(가파른 내리막) 이 오차가 눈덩이처럼 커지면서
속도가 실제보다 너무 빨리 깎여 **0을 지나쳐 마이너스로 뒤집혔다**. 한번
뒤집히면 중력항(고정 방향)과 마찰항(뒤집힌 방향으로 커짐)이 서로 같은
편이 되어 발산했다.

### 수정

```swift
if dt >= 0 {
    velocity += acceleration * dt
    applyRollingResistance(deltaTime: dt)
    position += velocity * dt
} else {
    position += velocity * dt   // 되돌리기 전(진입 시점) 속도로 위치 갱신
    applyRollingResistance(deltaTime: dt)
    velocity += acceleration * dt
}
```

검증: 60스텝 정방향 + 60스텝 역방향 라운드트립 테스트(10도 경사)에서
오차가 부동소수점 노이즈 수준(~1e-6)까지 줄어듦을 확인. 리뷰어가 1D
예제로 손으로 대수 재검증까지 완료.

## 5. "마이너스 속도"가 의미하는 것 — 두 가지 서로 다른 경우

세션 중 "마이너스 속도"가 왜/언제 나오는지 여러 번 논의됐다. 두 가지를
구분해야 한다.

### 경우 A: 계산 오류로 인한 마이너스 (위 4절의 버그)

정상적인 궤적 안에서, 아직 유효한 시점인데도 계산이 잘못돼서 속도가 조기에
0을 지나쳐버린 경우. 이건 버그이고 이미 고쳐졌다(4절).

### 경우 B: 물리적으로 실현 불가능한 목표 속도 (미해결, 아래 6절)

경사가 공 위치부터 홀컵을 지나서까지 쭉 일정하게 내리막이라고 하면, 공을
**정지 상태(속도 0)에서 놔줬을 때** 홀컵에 도달하는 속도가 물리적으로
가능한 **최소값**이다:

```
v_hole_min = √(2 · a_net · D)
  (a_net = accelMag - rollingResistance, D = 공-홀컵 거리, 등가속도 근사)
```

**목표 통과 속도(예: 0.05 m/s)가 이 `v_hole_min`보다 작으면, 그 경사/거리
조합으로는 절대 나올 수 없는 속도다** — 아무리 살살 놔도 이미 이보다 빠르게
도착하기 때문. 이 경우 역방향 계산이 공 위치 근처에서 마이너스 속도를
요구하는 것은 버그가 아니라 **"이 조합으로는 해가 없다"는 수학적으로
정확한 신호**다.

## 6. 마이너스 속도 처리: 시도했으나 기각됨 (경사 임계 + 물리적 해 없음 확인)

경우 B(위 5절)를 코드가 어떻게 처리할지에 대한 논의와 실제 시도 기록.

### 시도한 방식: 종료 지점에서 사후 보정

처음 확정했던 방향은 다음과 같았다:

> 마이너스 속도가 나오더라도 중간에 멈추지 말고, 원래의 종료 조건(공-홀컵
> 직선에 수직이고 공 위치를 지나는 선을 넘는 지점)까지 계산을 계속
> 진행시킨다. 그 최종 도달 지점에서 나온 속도 벡터의 방향을 구하고,
> 거기에 (마이너스를 0으로 만드는) 오프셋을 더해서 속도 크기만 보정한다.

즉 `if simd_length(ball.velocity) < 0.0001 { return nil }` 조기 포기 분기를
제거하고, 종료선 도달 시점에 `simd_dot(속도, initialDirection)`이 마이너스면
방향은 `initialDirection`을 유지하고 속도만 최소값(0.05)으로 보정하는 방식으로
실제 구현했다.

### 실제로 돌려본 결과: 종료선에 도달하는 경우 자체가 없었다

이 수정을 두 가지 시나리오로 테스트했다:

1. **10도(비현실적으로 가파른) 경사**: 홀컵에서 겨우 0.006m 떨어진 지점에서
   즉시 반사가 일어남.
2. **6%(현실적인 경계) 경사**: 계산상으로는 홀컵에서 약 0.65m 지점에서
   반사가 일어나야 했음.

**두 경우 모두 결과는 `nil`이었다** — 이유는 반사가 일어나는 순간
**위치의 진행 방향도 함께 뒤집히기 때문**이다. 위치 갱신이 `속도 × dt`로
계산되는데, `dt`는 항상 고정된 부호(음수)이므로 속도의 부호가 바뀌면
위치가 "공 쪽으로 나아가던 것"에서 "홀컵 쪽으로 다시 표류하는 것"으로
전환된다. 그 결과 시뮬레이션은 (a) 스캔된 지형 범위를 벗어나거나
(b) 스텝을 다 소진하거나 둘 중 하나로 끝나버리고, **종료선(공-홀컵
수직선)에 도달하는 경우 자체가 없었다** — 경사가 완만하든 극단적이든
마찬가지였다.

**결론**: "종료 지점에서 사후 보정"이라는 방식은, 목표 속도가 물리적
최소치보다 작은 모든 경우에 대해 **실행될 기회 자체가 없는 죽은 코드**였다.
이건 손으로 추론해서 예상한 게 아니라, 실제로 두 가지 다른 경사(10도,
6%)로 xcodebuild 테스트를 돌려서 직접 확인한 결과다.

### 대안(각도 탐색)도 기각됨 — 현실 데이터 기준으로 과한 엔지니어링

"종료선에 못 도달하면, 홀컵에서의 출발 방향 자체를 좌우로 틀어가며
재시도하자"(슈팅법)는 대안도 논의했다. 물리적으로는 타당한 접근이지만,
다음 실측 정보를 반영해 기각했다:

- **실제 골프 그린 경사의 90% 이상이 3% 미만**이고, 5%를 넘는 경우는
  거의 없다 (사용자 실측 정보).
- 기본 `rollingResistance`(0.35)에서 `accelMag == rollingResistance`가
  되는 경사각이 **정확히 약 2.87도(≈5% 경사) 근처**다 — 즉 "이 문제
  자체가 발생할 수 있는" 최소 조건이 이미 "거의 없는" 경사 구간에
  걸쳐 있다.
- 실제 홀컵 통과속도(0.1~0.3 m/s, 아래 참고)까지 감안하면, 이 문제는
  **현실적인 그린/퍼팅 조합에서는 사실상 발생하지 않는 극단적 경계
  케이스**다.

사용자의 표현을 그대로 빌리면: "경사도가 너무 높으면 만세를 불러야해
— 우린 못한다."

### 최종 결정: 원래대로 되돌리고, 대신 `holeCrossingSpeeds`만 현실화

- `backwardCandidate`는 원래의 `nil` 반환 동작으로 되돌렸다 (커밋
  `3800a83`). 이 케이스에서 `nil`이 나오는 것은 버그가 아니라 "이
  속도로는 이 지형에서 해가 없다"는 정확한 신호이므로, 이를 명시적으로
  검증하는 회귀 테스트
  (`testBackwardCandidateReturnsNilWhenCrossingSpeedBelowPhysicalMinimumOnSteepSlope`)
  를 추가해 향후 실수로 "고쳐야 할 버그"로 오인되지 않도록 문서화했다.
- 이번 조사에서 나온 진짜 유용한 개선은 따로 있었다: **실제 홀컵 통과속도가
  0.1~0.3 m/s**라는 사용자의 실측 정보를 반영해, 기존 `holeCrossingSpeeds`
  기본값(`[0.03, 0.05, 0.08, 0.11, 0.14]`, 대부분 이 범위보다 낮음)을
  `[0.1, 0.15, 0.2, 0.25, 0.3]`으로 조정했다. 같은 커밋(`3800a83`)에 포함.
- 골프 그린의 실측 물리 범위(경사 분포, 통과속도)는 향후 다른 튜닝에도
  참고할 수 있도록 전역 메모리(`~/.claude/projects/.../memory/

## 8b. 향후 아이디어 (미착수): 핀치 제스처로 레이캐스트 격자 크기 조절

Task 6에서 만든 3x3 지형 샘플링 격자(`terrainSampleGridResolution`,
`terrainSampleGridSpan`)는 지금 화면 비율(0.2~0.8) 고정이다. 논의된
아이디어: 두 손가락 핀치(와이드/내로잉)로 이 격자의 화면상 범위를
사용자가 직접 넓히거나 좁힐 수 있게 하자는 것 — 중심 위치는 고정하고
범위만 조절.

**구현 시 주의할 핵심 통찰(사용자 지적)**: 화면상 격자 크기는 고정이어도,
**카메라-지면 거리(원근 투영)에 따라 실제 지면에서의 격자 칸 크기는 계속
달라진다** — 카메라가 지면에 가까우면 같은 화면 비율이라도 실제 지면
범위는 좁고, 멀면 넓어진다. 그래서:

- 화면 격자 → 실제 지면 격자로 "대략 변환"되는 것이지, 고정 크기가 아니다.
- `terrainSampleMinSpacing`(현재 0.5m 고정 — 수집 버스트 간 최소 카메라
  이동 거리)도 이 실제 지면 격자 칸 크기에 연동시켜야 할 수 있다 — 안
  그러면 카메라가 지면에 가까울 때(격자가 촘촘할 때) 스캔 버스트 사이
  간격이 격자 칸보다 커서 빈틈이 생기거나, 반대로 멀 때(격자가 성길
  때) 불필요하게 자주 스캔할 수 있다.
- 즉 "격자 칸 2개 길이보다는 (수집 버스트 간 이동거리가) 더 커야 한다"는
  제약 자체가 카메라 거리에 따라 계속 바뀌는 값이라, 고정 상수로는 정확히
  맞추기 어렵다는 게 핵심 문제.

범위가 꽤 커서(제스처 인식 + 원근 투영 기반 동적 스케일링) 이번 시각화
작업(궤적 표시, 레이캐스트 마커)과는 분리해 별도 설계로 다루기로 결정.

**추가 통찰(사용자 제안)**: 화면 격자 크기로부터 실제 지면 격자 크기를
미리 예측(원근 투영 역산)하려 하지 말고, **이미 raycast로 얻은 9개 점의
실제 월드 좌표에서 인접한 두 점 사이 거리를 직접 측정**해서 그 값을 다음
`terrainSampleMinSpacing` 기준으로 쓰면 된다 — 매 스캔 버스트마다 "격자
2칸이 실제로 차지하는 거리"를 그 자리에서 재계산해서 다음 버스트를
트리거할 최소 이동거리로 갱신하는 방식. 카메라 각도/높이를 따로 추정할
필요 없이 이미 갖고 있는 raycast 결과로 바로 구해지므로, 고정 상수
(`terrainSampleMinSpacing = 0.5`) 대신 매번 동적으로 계산된 값을 쓰는
방향이 더 깔끔한 해법으로 보인다.

측정한 "격자 2칸 거리"를 그대로 다음 트리거 기준으로 쓰기보다는, 너무
가깝지도(중복 스캔) 너무 멀지도(빈틈 발생) 않도록 **배수 범위(예: 측정값의
2배 이상 4배 미만)로 다음 스캔을 트리거하는 밴드를 두자**는 추가 제안도
나왔다 — 정확한 배수 값은 실기기 테스트로 튜닝 필요.
