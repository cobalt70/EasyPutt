# CLAUDE.md — EasyPutt

EasyPutt 작업 시 참고할 프로젝트 지식과 작업 방식 피드백 모음.

## 프로젝트 지식

### 골프 그린의 물리적 한계
그린의 폭/길이는 사실상 ~30m를 넘지 않고, 현실적인 퍼팅 볼 스피드는 ~2-3 m/s(강한 퍼트)를 거의 넘지 않는다. 굴러가는 중 10 m/s는 이미 비현실적인 영역이다.

- `PuttRangeFinderConfig`의 상수(`holeCrossingSpeeds` 스윕 범위, `TerrainSampleStore.maxDistance`, `maxBackwardSteps`/`maxForwardSteps`, 지형 샘플 그리드 범위)를 튜닝하거나 계산된 후보 속도/방향의 타당성을 판단할 때, 볼-홀 거리 ~30m 초과나 롤링 스피드 ~3-4 m/s 초과는 이 앱의 현실적 도메인 밖으로 취급한다.
- 그 범위에서만 해가 나오는 후보/계산은 정답이 아니라 뭔가 잘못됐다는 신호다(예: 실현 불가능한 홀 통과 속도).

## 작업 방식 피드백

### 역방향 적분의 부호 규약 — 추상적 추론 금지
역시간 물리 적분(dt<0, `PuttRangeFinder.swift`에서 홀→볼 역추적)에서는 `position += velocity * dt`이므로 위치가 속도 벡터의 *반대* 방향으로 이동한다. 시드 속도를 `+angle`(`+rightAxis` 쪽)로 회전시키면 실제 *공간 경로*는 `-rightAxis` 쪽으로 휜다 — 순방향 적분 직관과 반대다.

- 실제로 `backwardOnlySolve`의 coarse-walk 탐색 방향(`searchSign`)이 뒤집혀 해에서 멀어지며 조용히 빈 결과를 반환하는 버그가 있었고, 추상적 부호 추론은 두 번 연속 틀렸다.
- 부호/방향 규약을 디버깅할 때는 "어느 쪽으로 밀려야 한다"를 추상적으로 따지지 말고, 구체적 좌표(예: hole=(0,0,0), ball=(0,0,5))를 대입해 실제 벡터 성분을 단계별로 계산해서 확인한다.

### 반복 계산보다 해석적 근사 우선
시스템의 다른 곳에 이미 보수적인 허용 오차가 있다면(예: `captureRadius` 3.3cm, 중심해 수렴 허용치 2.5cm), 더 정밀한 반복/이분 탐색으로 정확도를 쥐어짜는 건 낭비다.

- 좌/우 "makeable window" 경계는 물리 재시뮬레이션 대신 소각 근사(`atan(shiftDistance / distance)`)를 수렴된 중심해 방향 벡터에 직접 적용하는 것으로 충분했다. ("시뮬로 계산하면 너무 느리다는거지, analytic solution으로 가자")
- 반복법이 수렴하려는 목표가 이미 더 거친 현실 허용 오차로 묶여 있다면, 닫힌 형태의 근사를 먼저 제안한다.

### 확인 질문 최소화
작고 되돌리기 쉬운 변경(UI 레이아웃 조정, 버튼/인터랙션 재설계, 소규모 기능 추가)은 선택지 질문을 반복하지 말고, 합리적인 가정을 명시하고 바로 구현한 뒤 빌드 확인 후 보고한다. 사용자는 만들어진 결과를 보고 피드백하는 빠른 반복을 선호한다. 명시적 확인은 되돌리기 어렵거나 아키텍처에 영향이 큰 결정(데이터 모델 변경 등)에만 사용한다.
